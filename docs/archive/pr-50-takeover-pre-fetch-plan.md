## 2. 第一性原理

### 2.1 先保存事实，再提供能力

导入系统最重要的不是“能把数据塞进去”，而是“不改变外部数据表达的事实”。因此：

- [ ] 精确会话只有在来源确实提供开始和结束时间时才成立。
- [ ] 小时汇总只表达“某程序在某小时累计使用了多久”。
- [ ] 不通过顺序排列、平均分配或连续拼接生成来源中不存在的时间线。
- [ ] Patina 的 History 只展示精确会话；小时汇总只参与其支持的统计和排行。
- [ ] UI 按数据能力显示信息，缺失能力时隐藏对应入口或内容，而不是补假数据。

### 2.2 外部输入先转成稳定边界，再写核心数据

外部格式会变化，Patina 内部存储也会演进。两者不能直接耦合：

1. 外部解构器只理解来源格式。
2. 解构结果写成可检查的 Patina 通用 CSV。
3. 通用导入器只理解 Patina 通用 CSV。
4. 预览、校验通过后，事务性写入数据库。

对应检查项：

- [ ] 解构工具不会直接写 Patina 数据库。
- [ ] 解构输出放在源文件同级目录，使用 `.patina.csv` 后缀。
- [ ] 用户可以在导入前打开、检查、保存或修正通用 CSV。
- [ ] 通用导入只接受 Patina 定义的 CSV，不直接接受任意第三方 CSV/DB。
- [ ] 第三方 `.csv`、`.db` 由解构工具识别并转换；不支持的来源给出明确错误。

### 2.3 owner 先于文件位置

“导入”是数据边界和持久化能力，因此真实 owner 是 Rust `data` 层：

- [ ] 使用 `src-tauri/src/data/import/` 承载模型、通用 CSV、预览、提交和外部解构器。
- [ ] 使用 `data/repositories` 承载导入批次及其记录关联查询。
- [ ] `commands/import.rs` 只做参数接收、调用和错误映射。
- [ ] 不把新业务逻辑放进 `lib.rs`、`app/*`、`commands/*` 或 `shared/*`。
- [ ] 不通过 `backup` 模块复用与导入语义不同的恢复流程。

### 2.4 可撤销是导入可信度的一部分

导入数据不是本机追踪数据。用户必须能按某次导入精确撤销：

- [ ] 每次成功导入形成一个内部稳定批次 ID。
- [ ] UI 只显示当前批次列表的 `1、2、3……`，不暴露内部 ID。
- [ ] 删除中间批次后，显示编号按剩余顺序重新排列。
- [ ] 删除全部批次后，下一次导入重新显示为第 1 次。
- [ ] 删除批次只删除该批次导入的事实记录，不影响 Patina 原生记录。
- [ ] 删除操作在单个事务中完成；失败时整体回滚。
- [ ] 分类、应用改名和映射等用户资产不因删除批次被自动删除。

### 2.5 贡献归属服务于事实，而不是凌驾于质量

- [ ] 保留贡献者原始提交或至少保留其 author 信息。
- [ ] 维护者的重构、删减和补充使用独立提交，不把贡献者提交改署名为维护者。
- [ ] 最终合并不 squash，以便 Git 历史表达双方工作。
- [ ] 如果最终主体实现、UI、owner 和测试都需要由维护者重写，则不得仅为保留贡献痕迹而合并 PR。
- [ ] 即使当前 PR 不能直接合并，也要把确实沿用的贡献者代码、fixture 和测试整理为保留原 author 的来源提交；不能把有效贡献降格为只有文字致谢。
- [ ] 当前 intake 是否失败不决定是否保留有效贡献，只决定最终提交链能否进入合并。
- [ ] 在任何变基或强制更新前重新读取远端 head，避免覆盖贡献者的新提交。

## 5. 通用 CSV 合同的评审清单

字段名称可在实现时微调，但每条记录必须能表达以下事实：

- [ ] `record_type`：只允许 `exact_session` 或 `hour_bucket`。
- [ ] `start_time`：带时区或具有明确时区解释的开始时间。
- [ ] `duration_ms`：非负、有限、不会溢出的持续时长。
- [ ] `exe_name`：必填的可执行程序标识，例如 `Code.exe`。
- [ ] `end_time`：仅精确会话需要；小时汇总必须为空。
- [ ] `app_name`：可选的来源显示名，不覆盖 Patina 内的用户改名。
- [ ] `title`：可选；来源没有时留空。
- [ ] `path`：可选；来源没有时留空。
- [ ] `category`：可选的来源分类建议，不自动覆盖用户分类。
- [ ] `source` 或等价元数据：可说明解构器类型和格式版本，但不作为显示批次编号。

结构约束：

- [ ] `exact_session` 同时满足 `end_time > start_time`，且持续时长与二者差值在允许误差内一致。
- [ ] `hour_bucket` 的 `start_time` 对齐小时边界，`end_time` 为空，持续时长不超过来源格式允许的上限。
- [ ] 时间解析不依赖当前系统区域格式。
- [ ] 重复表头、BOM、空行、非法编码、巨大字段和未知列有明确处理策略。
- [ ] CSV 公式注入风险在导出/打开场景中被测试和说明。
- [ ] 解析错误包含行号、字段和可理解原因，但不泄露无关本机路径。

## 10. 停止条件与回滚策略

遇到以下任一情况时停止实施，不继续“顺手修”：

- [ ] PR head 在同步或推送前发生未知变化。
- [ ] 需要覆盖贡献者新提交或使用未授权的强制推送。
- [ ] Tai CSV 解析无法在不保留虚构会话逻辑的情况下独立出来。

- [ ] 需要新增跨层共享抽象、兼容壳或与架构文档冲突的新 owner。
- [ ] 最终主体实现已变为维护者重写，PR 不再满足外部贡献合入标准。
- [ ] 测试需要真实用户数据库且没有安全、脱敏、可重复的 fixture。

回滚原则：

- [ ] fetch 只新增/更新本地引用，可通过停止使用该引用恢复，不触碰工作树。
- [ ] 实施阶段每个提交保持聚焦，可用新增反向提交撤销；不使用破坏性 reset 清理用户工作。
- [ ] 数据迁移必须先定义降级/回滚路径，再允许进入正式数据库。
- [ ] 解构输出不修改源文件；失败时删除未完成的临时输出。

## 2. 第一性原理

### 2.1 迁移记录必须描述数据库事实

- [x] `_sqlx_migrations.success = 1` 只能表示对应表、列、约束和索引真实存在。
- [x] 不允许为了整理旧迁移历史而预先声明尚未执行的新迁移成功。
- [x] 迁移器必须先看到第 6 号迁移缺失，执行建表后才能记录成功。

### 2.2 新版本不采用旧结构兼容分支

- [x] 分类查询继续把 `import_time_buckets` 作为当前数据模型的一部分。
- [x] 不在前端使用“表不存在就只查 sessions”的降级 SQL。
- [x] 正确修复点是 Rust `data/sqlite_pool.rs` 的 schema 检查和迁移初始化链。

### 2.3 原生数据优先于功能恢复速度

- [x] 不直接修改用户数据库迁移记录。
- [x] 不删除、重建或复制 `sessions`。
- [x] 修复只补建缺失的空导入表与索引。
- [x] 实际启动验证前后分别核对 `sessions` 数量和 SQLite 完整性。

## 3. 已确认根因

- [x] 当前数据库有 57,312 条 `sessions`。
- [x] 当前数据库没有 `import_batches`、`import_exact_records`、`import_time_buckets`。
- [x] `_sqlx_migrations` 中第 6 号记录被标记成功，且 checksum 与当前迁移完全一致。
- [x] `normalize_current_baseline_migration_history_for_pool` 只判断到第 5 号 schema。
- [x] 当第 1 至 5 号结构存在时，该函数错误地把期望迁移列表保留到第 6 号并写入成功记录。
- [x] 随后的 SQLx migrator 因第 6 号已被记录成功而跳过 `IMPORT_DATA_SCHEMA_SQL`。
- [x] 分类查询访问不存在的 `import_time_buckets` 后失败，前端 catch 又将其呈现为空白冷状态。

# 全局与网页标题记录控制详细执行方案

> 文档类型：一次性 How-to 执行计划  
> 目标读者：Patina 维护者与后续实现代理  
> 当前状态：已完成  
> 完成去向：全部验收通过后移至 `docs/archive/`，不得长期留在 `docs/working/`

## 1. 目标

在不停止时间追踪、不删除任何既有记录的前提下，为标题采集建立一套可理解、可立即生效、入口一致的控制系统：

- 设置页提供“全局标题”开关。
- 托盘提供与全局开关等价的快捷动作，按当前状态显示“屏蔽标题”或“记录标题”。
- 分类页保留应用单项标题控制，并新增网页域名单项标题控制。
- 全局关闭时覆盖所有应用与网页单项设置，但不改写或丢失这些单项设置。
- 任一开关恢复后只记录未来产生的标题，不补写关闭期间的标题。

## 2. 第一性原理

### 2.1 标题与时间是两种不同的数据

时间追踪回答“用户在什么对象上使用了多久”，标题回答“当时具体查看了什么”。标题更敏感，也更细粒度，因此必须能够独立关闭。

由此得到不可违反的规则：

- 关闭标题记录不得停止应用时间或网页时间记录。
- 关闭标题记录不得停止应用识别、域名识别、URL、favicon 或分类逻辑。
- 标题开关不得复用“暂停追踪”或“排除统计”的状态。

### 2.2 开关控制未来写入，不控制历史存在

开关的职责是决定从当前边界开始是否允许写入标题，而不是清理数据库。

由此得到不可违反的规则：

- 关闭开关不删除历史标题、标题样本或历史网页段。
- 开启开关不回填关闭期间的标题。
- 历史数据清理若未来需要，应作为独立、高风险功能另行立项。

### 2.3 全局开关是总闸，单项开关是局部意愿

用户需要一个能立即覆盖全部标题采集的总闸，同时也需要保留各应用、各网页域名的长期偏好。

有效状态必须按下面的布尔规则计算：

```text
应用有效标题记录 = 全局标题记录开启 AND 应用单项标题记录开启

网页有效标题记录 = 全局标题记录开启
                   AND 网页活动记录开启
                   AND 域名时间记录开启
                   AND 域名单项标题记录开启
```

全局开关关闭时，只改变“有效状态”，不得把所有单项设置批量写成关闭。

### 2.4 用户操作必须在明确边界后立即可信

用户点击“屏蔽标题”后，不能继续因为缓存、活动会话或未封口网页段而把旧标题延长到关闭之后。

由此得到不可违反的规则：

- 设置保存成功或托盘操作成功后，运行时必须立即收到新状态。
- 当前应用标题样本必须在切换时间点封口；之后继续记录无标题时间。
- 当前网页标题段必须在切换时间点封口；之后继续记录同一域名的无标题段。
- 失败时不得只改变 UI；持久化、运行时与菜单文案必须保持一致或回滚到一致状态。

### 2.5 同一个状态只能有一个权威来源

设置页和托盘只是两个入口，不能各自维护一份状态。

权威状态应持久化为 settings 中的全局布尔值，并由 Rust 写侧负责原子更新与运行时通知。前端读取同一设置，托盘也读取同一设置。

## 4. 范围

### 4.1 本次包含

- 全局标题记录设置、默认值、序列化和持久化。
- Rust 运行时对全局标题状态的即时应用。
- 应用标题采样在全局或单项关闭时的正确封口。
- 网页活动写入链对全局和域名单项标题状态的统一裁决。
- 网页域名 override 的 `captureTitle` 契约闭环。
- 设置页、分类页和托盘的入口与同步。
- 中英文文案、可访问性状态、失败反馈。
- 单元测试、Rust 测试、集成回归和人工验收。
- 完成后的计划勾选、归档和 Project 状态建议。

### 4.2 本次不包含

- 删除或匿名化历史标题。
- 修改应用时间、网页时间或统计排除逻辑。
- 修改 URL、域名、favicon 的采集策略。
- 为浏览器扩展新增协议字段；标题裁决由 Patina 本机接收与写入侧完成。
- 新增标题白名单、正则过滤、按浏览器控制等扩展能力。
- 重做分类页或设置页布局。

## 5. 数据模型与兼容策略

### 5.1 全局设置

- [x] 在共享 `AppSettings` 增加 `titleRecordingEnabled: boolean`。
- [x] 在发布默认配置中设为 `true`。
- [x] 在前端 settings raw key 映射中增加 `title_recording_enabled`。
- [x] 在 Rust settings 读取/写入边界增加同名 key，并统一解析 `1/0`。
- [x] 缺少该 key 时按 `true` 处理，确保既有安装升级后行为不突变。
- [x] 检查备份、恢复和设置快照路径；确认该 key 随现有 settings 机制自然保存和恢复。
- [x] 不创建新表；该能力属于设置，不需要数据库结构迁移。

### 5.2 应用单项设置

- [x] 继续使用现有应用 override 的 `captureTitle`。
- [x] 明确 `captureTitle` 缺失或 `true` 均表示允许，只有显式 `false` 表示屏蔽。
- [x] 不批量重写既有应用 override。
- [x] 保持单项设置保存、取消和草稿脏状态语义不变。

### 5.3 网页域名单项设置

- [x] 在共享 `WebDomainOverride` 正式增加 `captureTitle?: boolean`。
- [x] 统一所有 normalize、compare、draft、save、load 路径：缺失等价于 `true`，仅在 `false` 时持久化关闭意图。
- [x] 修正当前已有部分 `captureTitle` 逻辑但共享类型未闭环的不一致。
- [x] 确认升级、备份恢复和旧 JSON 读取对未知/缺失字段安全。
- [x] 不要求修改浏览器扩展数据结构。

## 6. 运行时所有权与状态传播

### 6.1 建立单一写侧服务

- [x] 在 Rust 数据/服务边界提供读取与更新全局标题设置的明确入口。
- [x] 更新操作必须先可靠写入 settings，再更新运行时缓存和托盘菜单。
- [x] 暴露薄 Tauri command，供设置页保存路径调用；command 只做参数接收和转发。
- [x] 托盘复用同一服务，不复制写 settings 的逻辑。
- [x] 成功后发出统一的数据/设置变化事件，使已打开设置页和分类页能刷新。
- [x] 定义稳定 reason，例如 `title-recording-enabled` 与 `title-recording-disabled`，并覆盖测试。
- [x] 写入失败时保持原运行时状态和原托盘文案，并记录可诊断错误。

### 6.2 运行时缓存

- [x] 为全局标题状态建立有明确 owner 的运行时快照或缓存。
- [x] 启动时从 settings 初始化；读取失败按兼容默认值 `true`，同时记录错误。
- [x] 设置页/托盘成功写入后立即更新缓存，不等待轮询 TTL。
- [x] 单项应用和域名缓存不得越过全局总闸。
- [x] 对缓存失效、并发点击和重复设置相同值增加测试。

## 7. 应用标题写入链实施

- [x] 定位前台窗口采样进入 tracking runtime 的唯一裁决点。
- [x] 将有效判断改为“全局 AND 应用单项”，避免在多个调用点分别判断。
- [x] 全局或应用单项关闭时，在进入持久化前把新采样标题视为 `None`。
- [x] 切换关闭的时间点立即结束当前活动标题样本，后续同一 session 继续累计无标题时间。
- [x] 检查 `sessions.window_title` 兼容摘要字段：不得让关闭前的最后标题被解释为覆盖关闭后的整个 session。
- [x] 若该摘要仍服务旧读模型，应在关闭边界清除活动 session 的当前摘要值，但不得修改已封口历史 session。
- [x] 重新开启时，下一次真实窗口采样可创建新标题样本；不得恢复或延伸关闭前样本。
- [x] 验证应用切换、标题变化、锁屏、AFK、暂停追踪和退出封口的组合行为。

## 8. 网页标题写入链实施

- [x] 在本机网页活动接收后的 Rust 写侧加载全局标题状态和域名 override。
- [x] 在网页段合并/切分之前计算有效标题记录状态。
- [x] 有效状态关闭时只将 title 归一为 `None`；保留域名、URL、favicon、浏览器身份和时间。
- [x] 从有标题切换为无标题时，立即封口已有标题段，并从边界开始建立或延续无标题段。
- [x] 从无标题切换为有标题时，从下一条有效活动开始建立新标题段，不回填上一段。
- [x] 域名 `enabled=false` 继续按原逻辑停止网页时间；其优先级高于标题开关且语义不得混合。
- [x] 全局关闭时不改写 `WebDomainOverride.captureTitle`。
- [x] 覆盖同域名连续活动、跨域名、跨浏览器、心跳续期、重复 payload 和乱序 payload 测试。

## 12. 跨入口一致性

- [x] 设置页保存关闭：托盘立即显示 `记录标题`。
- [x] 设置页取消未保存修改：不得改变托盘或运行时。
- [x] 连续快速点击托盘：最终持久化值、运行时值和菜单文案一致。
- [x] 设置保存与托盘点击并发：按最终成功写入顺序收敛，不出现永久分裂。
- [x] 应用重启后，设置页、分类页和托盘均从同一持久化值恢复。
